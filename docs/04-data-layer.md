# 04 — Data Layer

The data layer is split across two packages:

- **`packages/db`** — Drizzle ORM schema, migrations, and the database client (Postgres or PGlite).
- **`packages/data`** — Provider REST clients, adapter orchestration, caching, failover, health tracking, and throttling.

---

## 1. Database Schema (20 Tables)

All tables are defined in `packages/db/src/schema/*.ts`. The schema uses Drizzle ORM with pg-core column types and is pulled together by a barrel export in `packages/db/src/schema/index.ts`.

### 1.1 Required Extensions

- **`pgcrypto`** — for `gen_random_uuid()` (UUID primary keys).
- **`vector`** — for pgvector embedding columns and HNSW indexes.

Declared in `_extensions.ts`; enforced by migration `0000_lazy_red_shift.sql`.

## 2. Database Client

**Location:** `packages/db/src/client.ts`

### 2.1 Connection

- **ORM:** Drizzle ORM (`drizzle-orm/postgres-js`).
- **Driver:** `postgres-js`.
- **URL:** Accepts `DATABASE_URL` or `POSTGRES_URL` (Supabase Vercel integration).
- **Instance:** Lazy singleton — `getDb()` initialises once per Node process and reuses across cold Vercel function invocations.

### 2.2 Pool Configuration

| Setting | Web (Vercel) | Worker | Override |
|---|---|---|---|
| `max` | 5 | 3 | `DB_POOL_MAX` / `WORKER_DB_POOL_MAX` |
| `prepare` | `false` | `false` | Required for Supabase transaction pooler |
| `idle_timeout` | 20s | 20s | |
| `connect_timeout` | 10s | 10s | |
| `max_lifetime` | 30 min | 30 min | Prevents long-lived connection buildup |

The runtime (`HAMAFX_RUNTIME=worker`) determines pool size: web defaults to 5 (to handle concurrent tool-call fan-out during chat turns), worker defaults to 3.

### 2.3 Local Development (PGlite)

When neither `DATABASE_URL` nor `POSTGRES_URL` is set, the system uses **PGlite** — an embedded Postgres that runs in-process.

- **Location:** `packages/db/src/pglite-client.ts`
- **Data:** Stored in `.hamafx/data/` (gitignored).
- **Migrations:** Applied automatically on first boot. The migration runner reads drizzle journal entries, parses SQL files, and executes them statement-by-statement.
- **pgvector handling:** Vector-specific SQL (CREATE EXTENSION, vector column types, HNSW indexes) is rewritten — vector columns become `real[]` and HNSW indexes are skipped. Any statement referencing `vector` or `hnsw` that fails is silently caught.
- **Unified entry point:** `getLocalDb()` in `packages/db/src/local-db.ts` auto-selects Postgres or PGlite.

### 2.4 Migrations

- 9 migrations (`0000` through `0008`) in `packages/db/drizzle/`.
- Production migrations are applied by the deploy/CI pipeline via `drizzle-kit migrate`.
- Local dev migrations are applied programmatically by PGlite on boot.

## 3. Providers

All provider code lives in `packages/data/src/providers/`. Each provider is a thin REST client that:
1. Self-throttles via `tryReserve()`.
2. Fetches from the upstream API with timeout and abort support.
3. Validates the response against a Zod schema.
4. Returns wire-shape objects or throws `ProviderError`.

### 3.1 BiQuote

**Role:** Primary provider for FX + XAU prices and OHLC candles.

**Endpoints:**
- `GET /api/{symbol}` — Latest tick for a single symbol.
- `GET /api/latest?symbols=XAUUSD,EURUSD,GBPUSD` — Batch ticks in one round trip.
- `GET /api/{symbol}/ohlc?interval=M1&limit=2000` — OHLC candles.

**Characteristics:**
- **No API key required.** Unauthenticated REST API.
- **Symbol mapping:** Identity (`XAUUSD` → `XAUUSD`). Whitelist enforced by `assertSupportedSymbol()` — only XAUUSD, EURUSD, GBPUSD.
- **Timeframe support:** M1 through D1. **No W1** (weekly) — BiQuote does not provide weekly bars; callers fall back to Finnhub.
- **OHLC response format:** `{ symbol, interval, bars: [...] }`. Bars return newest-first; the client reverses to oldest-first.
- **Candle cap:** 2000 bars per request.
- **Self-throttle:** 10 req/min (BiQuote is unauthenticated with no published fair-use cap). On 429, adaptive backoff drops to 80% of limit for 90s.
- **Timeout:** 8 seconds.
- **Soundness:** Monotonicity guard verifies ascending-time order after reversal.
- **SignalR:** A persistent SignalR hub connection for live ticks lives in `apps/worker/src/signalr/` (not in the provider package).

### 3.2 Finnhub

**Role:** Fallback for prices and candles. Primary for forex news.

**Base URL:** `https://finnhub.io/api/v1`

**Endpoints:**
- `GET /quote?symbol=OANDA:XAU_USD` — Latest price.
- `GET /forex/candle?symbol=...&resolution=...&from=...&to=...` — OHLC candles.
- `GET /news?category=forex` — Market news.

**Characteristics:**
- **Free tier:** 60 req/min total. Self-throttled at 30 req/min to leave headroom.
- **Symbol mapping:** `XAUUSD` → `OANDA:XAU_USD`, `EURUSD` → `OANDA:EUR_USD`, `GBPUSD` → `OANDA:GBP_USD`.
- **Resolution mapping:** Internal timeframes mapped to Finnhub resolution codes (1m→`1`, 5m→`5`, ..., 1d→`D`, 1w→`W`).
- **4H synthesis:** Finnhub's free `/forex/candle` does not natively support 4H. The client pulls 1H bars (4× the count plus buffer) and synthesises 4H bars by aggregating into UTC 4-hour bucket boundaries (00, 04, 08, 12, 16, 20).
- **History estimation:** `secondsForCount(tf, count)` computes the time window with a 25% buffer for weekend gaps.
- **News:** Up to 100 articles per call from the `forex` category. No entity tagging or sentiment — the AI agent infers sentiment from text.
- **Timeout:** 8 seconds.

### 3.3 Marketaux

**Role:** News fallback. Provides entity tagging and sentiment scores that Finnhub lacks.

**Base URL:** `https://api.marketaux.com`

**Characteristics:**
- **Free tier:** 100 req/day. Self-throttled at 4 req/min.
- **Endpoint:** `/v1/news/all` with search keywords covering XAU/gold, the three FX pairs, and macro keywords (Fed, FOMC, ECB, BoE, NFP, CPI, PCE).
- **Feature:** Returns `entities[]` with type, symbol, match_score, and sentiment_score. The `aggregateSentiment()` function weights per-entity sentiment by match_score to produce article-level sentiment labels.
- **Symbol extraction:** `extractSymbols()` unions explicit currency entities (`$USD`, `$EUR`, `$GBP`), pair patterns in title/snippet (`EUR/USD`, `EURUSD`), and gold keywords. Deduped and filtered to supported symbols.
- **Timeout:** 10 seconds.

### 3.4 FRED (Federal Reserve Economic Data)

**Role:** Macroeconomic calendar and actuals backfill. Intermarket resonance inputs.

**Base URL:** `https://api.stlouisfed.org`

**Endpoints:**
- `GET /fred/releases/dates` — Release schedules in a date window. Returns all release dates; filtered downstream to a curated set.
- `GET /fred/series/observations` — Historical series observations for actuals backfill.

**Characteristics:**
- **No formal quota** for occasional readers. Self-throttled at 30 req/min.
- **Curated releases:** 8 key US macro events mapped in `FRED_RELEASES`:
  - NFP (release 50, series PAYEMS)
  - CPI (release 10, series CPIAUCSL)
  - PCE (release 21, series PCEPI)
  - PPI (release 46, series PPIACO)
  - GDP (release 53, series GDP)
  - Retail Sales (release 86, series RSAFS)
  - Industrial Production (release 20, series INDPRO)
  - FOMC Decision (release 101, series DFEDTARU)
- **Date handling:** FRED returns date-only strings; the calendar adapter assumes 13:30 UTC (8:30 AM ET) for US releases.
- **Actuals backfill:** Patches `actual` values on `economic_events` rows where `actuals_filled_at IS NULL`.
- **Resonance:** `fetchResonanceInputs()` pulls DFII10 (10-Year Real Yields) and T10YIE (Breakeven Inflation) in parallel for the intermarket resonance computation.
- **Timeout:** 10 seconds.

### 3.5 CFTC

**Role:** Commitment of Traders weekly data.

**Base URL:** `https://publicreporting.cftc.gov/resource/gpe5-46if.json`

**Characteristics:**
- **No app token required** at our volumes (1 weekly call per symbol).
- **Dataset:** Disaggregated Futures-Only report (Socrata endpoint).
- **Query:** `$where=market_and_exchange_names="..."` with `$order=report_date_as_yyyy_mm_dd DESC` and `$limit=8`.
- **Commodity mapping:** `XAUUSD` → `GOLD - COMMODITY EXCHANGE INC.`, `EURUSD` → `EURO FX - CHICAGO MERCANTILE EXCHANGE`, `GBPUSD` → `BRITISH POUND - CHICAGO MERCANTILE EXCHANGE`.
- **Parsing:** String-typed integer columns parsed via `parseCftcInt()`; empty strings and `.` map to null.
- **Self-throttle:** 30 req/min.
- **Timeout:** 8 seconds.

### 3.6 Pseudo-Providers (Phase 8)

#### `live-ticks`

Reads the `live_ticks` Postgres table. Worker writes; Vercel reads.

- **Freshness:** Row must have `ts >= now - 5000ms`. If stale or missing, throws `ProviderEmptyError` so failover falls through to BiQuote REST **without** recording a health failure.
- **Pinned:** Always first in price failover order regardless of recent health score.

#### `candles-1m`

Reads the `candles_1m` Postgres table. Only relevant for 1-minute timeframe candles.

- **Freshness:** Newest bar must be within 90s of now (1.5× bar size).
- **Pinned:** Always first for 1m candles in failover order.

## 4. Adapters

Adapters live in `packages/data/src/adapters/` and are the public surface that routes/tools import. They orchestrate providers, caching, and failover.

### 4.1 Price Adapter (`price.ts`)

**Functions:** `getPrice()`, `getPriceWithMeta()`

**Failover Chain (3-tier):**
1. `live-ticks` (pinned) — Postgres snapshot, zero outbound HTTP. Returns `ageMs` from worker observation.
2. `biquote` — REST fallback. Uses server-computed `mid` (BiQuote suppresses `last` for FX).
3. `finnhub` — Last resort.

**Cache:** `PRICE_TTL` = 3s soft TTL, 10s max stale.

**Synthesised spread:** Providers on free tiers do not expose bid/ask. The adapter synthesises a zero-spread Tick (bid=ask=mid) so downstream chart/UI code does not special-case nullable bid/ask.

**Freshness metadata:** `PriceResult.stale` flag + `producedAt` ms epoch + `ageMs` (only meaningful from live-ticks).

### 4.2 Candles Adapter (`candles.ts`)

**Functions:** `getCandles()`, `getCandlesWithMeta()`

**Failover by Timeframe:**

| TF | Priority | Notes |
|---|---|---|
| 1m | candles-1m → biquote → finnhub | Worker-maintained 1m bars first |
| 5m–1d | biquote → finnhub | BiQuote covers all sub-daily |
| 1w | finnhub | BiQuote does not support weekly |

**Candle cap:** BiQuote capped at 2000 bars; Finnhub supports up to 5000. The adapter requests the minimum of requested count and provider cap.

**Cache:** `candleTtl(tf, lastBar)`:
- Closed-bar only (historical): 10 min TTL, 1 day max stale.
- 1m with last bar: 5s TTL, 1 min max stale.
- Other with last bar: 30s TTL, 5 min max stale.

**Soundness:** Partial data returned as-is. On all-provider failure the most actionable error is re-thrown.

### 4.3 News Adapter (`news.ts`)

**Functions:** `fetchNews()`, `articleIdFromUrl()`

**Failover:** Finnhub (primary, 60 req/min, `category=forex`) → Marketaux (fallback, 100 req/day, entity tagging + sentiment).

**Article deduplication:** `id = sha1(url)` — stable across providers.

**Symbol extraction (Finnhub):** Regex-based keyword matching on headline, summary, and related fields. Maps gold, EUR/USD, GBP/USD, Fed/FOMC/NFP/CPI/PCE, ECB, BoE keywords to our internal symbol/currency tags.

**Post-filter:** Optional `symbol` filter applied after mapping so one fetch benefits multiple symbols.

### 4.4 Calendar Adapter (`calendar.ts`)

**Functions:** `fetchUpcomingEvents()`

**Pipeline:** FRED release dates in a 14-day window → filter to curated `FRED_RELEASES` set → map to `EconomicEvent` with stable id `fred:<release_id>:<YYYY-MM-DD>`.

**FRED only:** No forecast/actual values at fetch time (backfilled later by the FRED actuals cron).

## 5. Caching

**Location:** `packages/data/src/cache/`

### 5.1 Cache Interface

```typescript
interface Cache {
  fetch<T>(key: string, ttlSeconds: number, producer: () => Promise<T>, tags?: string[]): Promise<T>;
  fetchWithMeta<T>(key: string, producer: () => Promise<T>, options: CacheFetchOptions): Promise<{ value: T; meta: CacheEntryMeta }>;
  invalidateTag?(tag: string): Promise<void> | void;
}
```

**Features:**
- **Single-flight deduplication:** Concurrent callers share the same in-flight producer promise.
- **Stale-while-revalidate/error (SWR):** When a producer fails and `maxStaleSeconds > 0`, the most recent cached value is served up to `expiresAt + maxStaleSeconds`. The adapter receives `meta.stale = true`.
- **Tag-based invalidation:** Resources are tagged (e.g. `hfx:price:XAUUSD`) for grouped revalidation.

### 5.2 Implementations

#### Next.js Data Cache (`nextjs.ts`)

Used inside Next.js request handlers on Vercel. The `NextjsCache` wraps a `MemoryCache` for SWR + single-flight (the in-memory tree owns the producedAt + hard-expiry math). Tag invalidation propagates through Next's `revalidateTag()`.

#### MemoryCache (`memory.ts`)

Process-local LRU-like map. Used in tests, scripts, and any non-Next context. Not shared across Vercel instances — the upstream provider throttle, not this cache, is what protects the provider from over-call.

#### Auto-selection (`cache/index.ts`)

`getDefaultCache()` checks `NEXT_RUNTIME` / `NEXT_PHASE` and dynamically imports Next.js cache. Falls back to `MemoryCache`. The dynamic import keeps `next` an optional peer dependency.

### 5.3 Cache Keys

**Scheme:** `hfx:<resource>:<symbol|null>:<tf|null>:<extra>`

Example: `hfx:candles:XAUUSD:1h:n300`

Resources: `price`, `candles`, `indicator`, `news-list`, `news-article`, `calendar-day`, `calendar-week`, `fred-series`.

Null slots use `_` as placeholder.

### 5.4 TTL Policies

| Resource | Soft TTL | Max Stale |
|---|---|---|
| Price | 3s | 10s |
| Candles (closed bars) | 10 min | 1 day |
| Candles (1m, last bar) | 5s | 1 min |
| Candles (other, last bar) | 30s | 5 min |
| News list | 60s | 10 min |
| News article | 24h | 7 days |
| Calendar day | 5 min | 1h |
| Calendar week | 15 min | 6h |
| FRED series | 6h | 7 days |

## 6. Failover

**Location:** `packages/data/src/failover.ts`

### 6.1 `runWithFailover<T>(attempts: ProviderAttempt<T>[])`

Generic primary-then-fallback runner:

1. **Sort providers** by health score (descending), with pinned providers keeping their caller-specified position.
2. **Try each provider** in order.
3. **On `ProviderError`:** Log, record health failure, continue to next.
4. **On `ProviderEmptyError`:** Log, skip health write, continue. This is the sentinel for "nothing to offer right now" — distinct from failure.
5. **On success:** Record health success, return `{ value, provider }`.
6. **On all failure:** Re-throw the most **actionable** error:
   - `PROVIDER_QUOTA_EXCEEDED` (rank 3) — strongest operator signal.
   - `PROVIDER_HTTP_ERROR` / `PROVIDER_TIMEOUT` (rank 2) — transient but debuggable.
   - `PROVIDER_PARSE_ERROR` / other (rank 1) — least actionable.
7. **On non-`ProviderError`:** Record failure and immediately rethrow (bug, not provider issue).

### 6.2 Provider Attempts

```typescript
interface ProviderAttempt<T> {
  name: string;
  pinned?: boolean;  // Keep caller-specified position regardless of health
  run(): Promise<T>;
}
```

Pinned providers (e.g. `live-ticks`, `candles-1m`) keep their position even when their health score temporarily dips — preventing a transient empty result during worker restart from permanently demoting the SignalR pipeline below the REST fallback.

### 6.3 Health System

**Location:** `packages/data/src/health.ts`

Per-provider rolling health using a circular buffer of success/failure samples within a 5-minute window.

- **`recordSuccess(provider)`** — append a success sample.
- **`recordFailure(provider)`** — append a failure sample.
- **`getHealth(provider)`** — returns `{ samples, ok, failed, failureRate }`.
- **`getScore(provider)`** — returns 0.5 for unknown providers (neutral — give them a chance), `1 - failureRate` for known providers (clamped to `[0, 1]`).

**Scope:** Per-Lambda-instance. Not shared across Vercel replicas. The pinned mechanism is the production lever for ordering guarantees; scoring is only the tiebreaker between unpinned alternatives.

### 6.4 Error Types

**`ProviderError`** — normalized upstream failure.

| Code | Description |
|---|---|
| `PROVIDER_TIMEOUT` | Request timed out (≥8s) |
| `PROVIDER_HTTP_ERROR` | Non-2xx response (not 429) |
| `PROVIDER_PARSE_ERROR` | Response did not match expected schema |
| `PROVIDER_QUOTA_EXCEEDED` | Rate limit hit (HTTP 429 or self-throttle) |
| `NO_PROVIDER_AVAILABLE` | No providers configured for the resource |

**`ProviderEmptyError`** — sentinel for "no data right now" (not a failure). Thrown by `live-ticks` and `candles-1m` when the worker hasn't flushed recently. Bypasses health-failure recording.

## 7. Throttling

**Location:** `packages/data/src/cache/throttle.ts`

### 7.1 Adaptive Rate Limiting

Each provider has a `ThrottleConfig`:

```typescript
interface ThrottleConfig {
  limit: number;           // Max calls per window
  windowMs: number;        // Window length in ms
  backoffFraction?: number; // Default 0.8 — reduce cap to 80% on backoff
  cooloffMs?: number;      // Default 90s — how long reduced cap stays
}
```

### 7.2 `tryReserve(provider, config)`

Reserve one call against a provider's quota. Returns `true` if allowed, `false` if capped.

**Two backends:**

1. **In-memory (default):** A `Map<string, Bucket>` tracking count, window start, and backoff. When the window expires, the count resets to 1. On 429 response, `noteBackoff()` sets `backoffUntil`; the effective limit drops to `floor(limit * backoffFraction)` until `cooloffMs` elapses.

2. **Postgres (`THROTTLE_BACKEND=postgres`):** Uses the `provider_throttle` table. The same window-roll and backoff logic runs in SQL:

```sql
INSERT INTO provider_throttle ... ON CONFLICT (provider) DO UPDATE SET
  count = CASE WHEN window expired THEN 1 ELSE count + 1 END,
  window_started_at = CASE WHEN window expired THEN now ELSE window_started_at END
WHERE (window expired) OR (count < effective_limit)
```

Returns a row only when the reservation succeeds; empty result means throttled. This backend shares the counter across all Vercel instances.

### 7.3 Provider Throttle Limits

| Provider | Limit | Window | Notes |
|---|---|---|---|
| Finnhub | 30/min | 60s | Free tier is 60/min; leave 50% headroom |
| BiQuote | 10/min | 60s | Unauthenticated; conservative cap |
| Marketaux | 4/min | 60s | Free tier is 100/day; prevent burst exhaustion |
| FRED | 30/min | 60s | No formal quota |
| CFTC | 30/min | 60s | 1 call/symbol/week is well under this |

## 10. Architecture Decisions

1. **Snapshot `live_ticks`, not tick history.** Tick history would balloon Supabase Free's 500 MB budget for marginal value — 1m candles cover replay use cases and the snapshot table has 3 rows.

2. **ProviderEmptyError ≠ health failure.** A transient empty result during worker restart is normal. The sentinel error type prevents health-score demotion that would permanently sideline the SignalR pipeline.

3. **Pinned providers for ordering guarantees.** `live-ticks` and `candles-1m` are pinned so they always go first regardless of health score. Health scoring is a tiebreaker between unpinned alternatives (e.g. biquote vs finnhub).

4. **No cross-instance cache.** The in-memory cache is per-process. The provider throttle, not the cache, protects against over-calling upstream APIs. If cross-instance cache consistency becomes needed, a Redis-backed `Cache` implementation can be swapped in behind the same interface.

5. **Postgres throttle is opt-in.** Default is in-memory (fast, zero infra). Set `THROTTLE_BACKEND=postgres` when multiple instances need a shared counter.

6. **No body text storage for news.** Only title + summary is stored. Users click through to the source for full articles. Keeps the database lean.

7. **FRED dates are date-only.** The calendar adapter assumes 13:30 UTC for US releases. This is a known approximation (some releases are at 10:00 ET, others at 8:30 ET).

8. **Twelve Data retired (Phase 8 PR-19).** After two weeks of soak with BiQuote as primary, chat telemetry showed Twelve Data was never selected. The dependency, API key, and adapter were removed.
