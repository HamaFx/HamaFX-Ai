# Implementation Plan: System Cleanup & Stability

This document outlines the system cleanup and stability improvements required for transitioning HamaFX-Ai from a single-user personal trading copilot into a robust, multi-user platform.

---

## 1. Dead Code & Unused Packages

To reduce build times, security surface area, and maintenance overhead, we will purge unused subsystems that were either placeholders or abandoned features.

- **Remove Directories:**
  - `packages/web3/` (empty placeholder)
  - `packages/worker-core/` (empty placeholder)
  - `contracts/` (smart contract stubs, unused)
- **Dependency Audit:**
  - Run `pnpm un [...]` to remove dependencies strictly tied to removed directories.
  - Remove references to these packages from `pnpm-workspace.yaml`.
  - Audit all remaining `package.json` files for unused dependencies using `pnpm dlx depcheck`.
- **Dead Exports:**
  - Audit exported functions across `packages/*` using tools like `ts-prune` or `knip` to find and eliminate dead code paths.

---

## 2. Environment Variable Cleanup

Currently, HamaFX-Ai relies on a single `APP_PASSWORD` for global access and uses scattered environment validators. We will centralize and properly validate all multi-user environment variables.

- **Variables to Remove:**
  - `APP_PASSWORD`
- **Variables to Add:**
  - `NEXTAUTH_SECRET` (Required)
  - `NEXTAUTH_URL` / `AUTH_URL` (Required)
  - `ENCRYPTION_SECRET` (For BYOK encryption at rest, Required)
  - `REDIS_URL` (Optional, for advanced rate limiting/caching)
- **Validation Consolidation:**
  - Merge redundant validation logic (e.g., `getAuthEnv` vs `getServerEnv`).
  - Move all environment parsing to `packages/shared/src/env.ts` using Zod, exposing a strongly-typed `env` object.
- **Source of Truth:**
  - Ensure `.env.example` is fully updated with clear comments delineating **Required** vs **Optional** variables.

```typescript
// packages/shared/src/env.ts
import { z } from "zod";

const envSchema = z.object({
  DATABASE_URL: z.string().url(),
  NEXTAUTH_SECRET: z.string().min(32),
  ENCRYPTION_SECRET: z.string().min(32),
  // Optional features
  REDIS_URL: z.string().url().optional(),
});
export const env = envSchema.parse(process.env);
```

## 3. Error Handling Improvements

Standardized error handling is critical when moving to a multi-user API where clients need predictable error formats to drive UI states.

- **Standard Response Format:**
  Enforce a standard JSON shape for all API error responses.
  ```typescript
  {
    "success": false,
    "error": {
      "code": "UNAUTHORIZED",
      "message": "You do not have access to this resource.",
      "details": {} 
    }
  }
  ```
- **Error Taxonomy:**
  Ensure all errors map to standardized codes (`NOT_FOUND`, `FORBIDDEN`, `VALIDATION_ERROR`, `RATE_LIMITED`).
- **React Error Boundaries:**
  Wrap major Next.js layouts in global and granular error boundaries to gracefully handle multi-user scenarios (e.g., user is deleted, session expires).
- **Security Codes (403 vs 401):**
  - Add specific 403 Forbidden handlers to catch Cross-User Data Access (IDOR) attempts.
- **PGlite Compatibility:**
  - Ensure database error handlers gracefully process dialect differences if running in PGlite mode vs standard Postgres.

---

## 4. Logging Improvements

Single-user apps often log globally. In a multi-user app, logs must be scoped, traceable, and privacy-conscious.

- **Structured Logging:**
  Implement structured JSON logging (e.g., using Pino).
- **Context Injection:**
  Ensure `userId` and `workspaceId` (if applicable) are injected into every log entry generated by authenticated requests.
- **Correlation IDs:**
  Propagate `X-Request-Id` across the web app, Next.js API, and worker processes.
- **PII Masking:**
  Use log redaction middleware to mask sensitive fields (e.g., `email`, `tokens`, `keys`).
- **Worker Logging:**
  Ensure background jobs (e.g., per-user briefings, MT5 polling) log execution contexts mapped to the executing `userId`.

---

## 5. Performance Improvements

Multi-user scaling introduces database contention and repetitive data fetching.

- **Connection Pooling:**
  Configure PgBouncer or update Drizzle/Postgres connection pool settings in the background worker to handle high concurrent user load.
- **Query Optimizations (N+1):**
  Audit ORM queries for N+1 issues, particularly around `user -> watchlists -> instruments`. Utilize DB joins or Drizzle's relational queries efficiently.
- **Caching Strategy:**
  Add lightweight caching mechanisms (in-memory or Redis) for high-read, low-write per-user data (e.g., active user settings, feature flags).
- **Index Review:**
  Add B-tree indexes to all `user_id` columns across the database.
  ```sql
  CREATE INDEX idx_watchlists_user_id ON watchlists(user_id);
  ```
- **Query Timeouts:**
  Add global statement timeouts (`statement_timeout`) to prevent rogue queries from taking down the instance.

---

## 6. Security Hardening

- **IDOR Audit:**
  Ensure every REST/GraphQL or Server Action endpoint strictly includes `WHERE user_id = current_user_id`.
- **Input Sanitization:**
  Validate and sanitize user-generated inputs (e.g., chat payloads, prompt names) using DOMPurify or equivalent to prevent Stored XSS.
- **Security Headers:**
  Implement standard security headers in Next.js (`next.config.js`), adjusting Content Security Policy (CSP) for the multi-user deployment model.
- **Rate Limiting:**
  Move from in-memory IP-based rate limiting to user-ID-based rate limiting on all mutation and AI-generation endpoints.
- **Data Encryption:**
  Ensure BYOK (Bring Your Own Key) secrets and exchange API keys are encrypted at rest using `ENCRYPTION_SECRET`.
- **Audit Logs:**
  Introduce an `audit_logs` table to track sensitive operations (key rotations, profile deletions).

---

## 7. Code Quality

- **Comment Purge:**
  Run project-wide search to eliminate legacy comments like `"// TODO: Fix for multi-user later"` or `"// single user mode only"`.
- **JSDoc Updates:**
  Update service-level functions with JSDocs defining `userId` expectations.
- **TypeScript Strictness:**
  Eliminate any lingering `any` types in database schemas, Next.js handlers, and AI tool interfaces.
- **API Cleanliness:**
  Use explicit `index.ts` barrels in `packages/*` to ensure clean public API boundaries.

---

## 8. Database Cleanup

Ensure database integrity handles multiple parallel users robustly.

- **Foreign Key Constraints:**
  Ensure all relations explicitly define constraints.
- **CASCADE Deletions:**
  Validate that `ON DELETE CASCADE` is set on tables depending on `users` so that deleting a user reliably purges their data.
- **CHECK Constraints:**
  Enforce valid enum states at the DB level, not just the ORM level.
- **Soft Deletes:**
  Implement `deleted_at TIMESTAMP` for user accounts to allow account recovery workflows and prevent accidental catastrophic data loss.
  ```sql
  ALTER TABLE users ADD COLUMN deleted_at TIMESTAMP DEFAULT NULL;
  ```

---

## 9. Docker & Deployment

Update local development and production containers for multi-tenant reality.

- **Multi-Stage Dockerfile:**
  Ensure the Dockerfile efficiently builds and separates `apps/web` and `apps/worker` artifacts.
- **Compose Security:**
  Remove hardcoded database credentials from `docker-compose.yml`. Use `.env` variables natively.
- **Production Compose:**
  Create a `docker-compose.prod.yml` configured for restart policies, resource limits (CPU/RAM), and private networking.
- **Health Checks:**
  Implement `/api/health` in Next.js and a health endpoint in the worker for Docker orchestrators to monitor.
- **Graceful Shutdown:**
  Ensure Node.js captures `SIGTERM`/`SIGINT` to close DB connections safely.

---

## 10. Dependency Updates

- **Audit Packages:**

- **Pinning:**
  Pin critical dependencies (Next.js, Drizzle, React) to avoid upstream breakages.
- **Dependabot:**
